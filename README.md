# PRSTNK — сайт

Статический сайт-магазин авторской графики. Управляется **данными**: каталог, художники и подборки лежат в `data/`, а готовые HTML-страницы собираются генератором `build.py`. Заявки идут в Telegram.

## Структура

```
prstnk-site/
├── data/                  ← ★ ИСТОЧНИК ПРАВДЫ (правишь это)
│   ├── artworks/<slug>.json   — по файлу на каждую работу (20 шт)
│   ├── artists/<slug>.json    — по файлу на художника (12 шт)
│   └── collections/<slug>.json — подборки (4 шт)
│
├── build.py               ← генератор: data/*.json → готовые HTML-страницы
├── admin/                 ← визуальная админка (Sveltia CMS) — форма вместо JSON
│   ├── index.html
│   └── config.yml
├── .github/workflows/     ← авто-сборка: правка данных → пересборка страниц
│
│   ── СГЕНЕРИРОВАННЫЕ (не править руками — перезапишутся!) ──
├── index.html             ← главная
├── works.html             ← каталог (бейджи статусов + фильтр по технике)
├── work-<slug>.html       ← страница каждой работы (20 шт, свой SEO + OG)
├── artists.html           ← индекс художников
├── artist-<slug>.html     ← страница каждого художника (12 шт)
├── journal.html           ← журнал «Во дела»
├── zine-04.html           ← полный выпуск № 04
├── lot.html               ← редирект на каталог (legacy)
├── sitemap.xml            ← карта сайта (37 URL)
│
│   ── НЕИЗМЕННЫЕ ──
├── prstnk.css             ← все стили + responsive + статусы/бейджи
├── prstnk.js              ← Telegram-ссылки, меню, фильтры, анимации
├── fonts.css / fonts/     ← локальные шрифты (44 .woff2)
├── favicon.svg / og-cover.svg / robots.txt
└── images/works/          ← сюда CMS грузит фото работ
```

## Как управлять каталогом

Сайт **data-driven**: страницы работ и художников НЕ редактируются руками — они собираются из `data/`.

### Способ A — визуальная админка (рекомендуется)

1. Открой `https://bregeadmin.github.io/prstnk/admin/`
2. Войди через GitHub *(разовая настройка авторизации — см. ниже)*
3. Редактируй работы/художников/подборки через форму: загрузка фото кнопкой, статус из списка, тип работы (тиражная/уникальная) переключателем
4. Сохрани → изменения коммитятся в GitHub → GitHub Action **сам пересобирает** страницы

## Запуск админки (разовая настройка, ~20 минут)

Админка живёт по адресу `https://bregeadmin.github.io/prstnk/admin/`. Чтобы в неё можно было войти и сохранять изменения, нужен бесплатный «пропускной пункт» — он проверяет вход через GitHub. Настраивается один раз. Всё бесплатно.

### Шаг 1. Завести Cloudflare и поднять «пропускной пункт»

1. Зарегистрируйся на [cloudflare.com](https://dash.cloudflare.com/sign-up) (бесплатно).
2. Открой [github.com/sveltia/sveltia-cms-auth](https://github.com/sveltia/sveltia-cms-auth) и нажми кнопку **«Deploy to Cloudflare Workers»** в описании. Пройди мастер.
3. После деплоя скопируй **адрес воркера** — вида `https://sveltia-cms-auth.ИМЯ.workers.dev`. Запиши его.

### Шаг 2. Создать «ключ доступа» (GitHub OAuth App)

1. Открой [github.com/settings/applications/new](https://github.com/settings/applications/new).
2. Заполни:
   - **Application name:** `PRSTNK админка`
   - **Homepage URL:** `https://bregeadmin.github.io/prstnk/`
   - **Authorization callback URL:** `<адрес воркера>/callback` (например `https://sveltia-cms-auth.имя.workers.dev/callback`)
3. Нажми **Register application**. Скопируй **Client ID**. Нажми **Generate a new client secret** — скопируй **secret** (показывается один раз!).

### Шаг 3. Связать воркер с ключом

В Cloudflare → твой воркер `sveltia-cms-auth` → **Settings → Variables and Secrets**, добавь:

| Имя | Значение |
|---|---|
| `GITHUB_CLIENT_ID` | Client ID из шага 2 |
| `GITHUB_CLIENT_SECRET` | secret из шага 2 → нажми **Encrypt** |
| `ALLOWED_DOMAINS` | `bregeadmin.github.io` |

Сохрани (Deploy).

### Шаг 4. Прописать адрес воркера в сайт

В файле `admin/config.yml` замени строку `base_url:` на свой адрес воркера:
```yaml
  base_url: https://sveltia-cms-auth.ИМЯ.workers.dev
```
Закоммить. Готово — на `…/prstnk/admin/` появится кнопка «Войти через GitHub».

### Шаг 5. Дать доступ партнёру

Чтобы партнёр мог **сохранять** работы (а не только смотреть):

1. У партнёра должен быть аккаунт на GitHub (бесплатный, регистрируется за минуту).
2. В репозитории: **Settings → Collaborators → Add people** → впиши его ник или email-приглашение.
3. Партнёр принимает приглашение из почты.

Теперь партнёр заходит на `https://bregeadmin.github.io/prstnk/admin/`, жмёт «Войти через GitHub» своим аккаунтом — и редактирует каталог через форму. Сохранил → GitHub Action сам пересобирает страницы → сайт обновился через 1–2 минуты.

> Совет: добавь и себя, и партнёра как Collaborators. Свой основной аккаунт-владелец репозитория тоже может входить в админку без приглашения.

> Доступность из РФ: `github.com` и `*.workers.dev` обычно открываются без VPN. Сам воркер и движок админки могут чуть медленно грузить интерфейс — для разовых правок это терпимо. Витрину покупателей это не касается вообще.

### Способ B — правка JSON напрямую

Пока работ мало, можно редактировать файлы `data/artworks/<slug>.json` прямо на GitHub (кнопка ✏️). После коммита Action пересоберёт страницы. Локально — `python3 build.py`.

### Тип работы: тиражная или уникальная

В каждой работе есть поле `workType`:
- **`editioned`** — тиражная. Заполняются `editionTotal` (всего), `editionNumber` (этот экз.), `availableCount` (доступно), `artistProofs`, `editionClosed`. Карточка показывает «ТИРАЖ · 30 / 14 ИЗ 30 / Доступно 16 экз.». Если `availableCount: 1` → «Последний экземпляр». Если `editionClosed: true` → «Тираж закрыт».
- **`unique`** — уникальная (1/1). Тираж не нужен, заполняется `uniquenessNote`. Карточка показывает «УНИКАЛЬНАЯ РАБОТА / 1 ИЗ 1».

Поле `status`: `available` / `reserved` (бронь) / `sold` (продано) — управляет бейджем в каталоге и кнопкой на странице.

### Фото работ

Пока `mainImage` пустой — показывается графическая заглушка (SVG). Загрузи скан через CMS (или положи в `images/works/` и впиши путь в `mainImage`) — заглушка заменится на фото.

## Что обязательно поменять перед запуском

### 1. Telegram-юзернейм — **в одном месте**

В `prstnk.js`, **первая константа**:

```js
const TG_USER = 'prstnk_store';
```

Меняешь — и все 20+ Telegram-кнопок и ссылок на обеих страницах автоматически переключаются. Тексты сообщений в кнопках задаются через `data-tg-text="..."` в HTML — править там же если надо изменить формулировку.

### 2. Контакты в подвале

В `index.html` и `lot.html` найди блок `<h4>Связь</h4>` — там сейчас плейсхолдеры:

```
hello@prstnk.ru
+7 812 000 00 00
наб. Обводного, 60
пн–сб 12:00–20:00
```

Поменяй на реальные. Email и телефон обёрнуты в `<a href="mailto:...">` / `<a href="tel:...">` — поменяй и сам href тоже.

### 3. Социальные сети в подвале

Footer: `<a href="https://instagram.com/prstnk_store">` и `<a href="https://vk.com/prstnk_store">`. Если в Instagram/VK другой ник — поменяй вручную. Telegram-ссылка генерируется автоматически из `TG_USER`.

Если каких-то соцсетей нет — удали соответствующую `<a>...</a>` строку из футера.

### 4. Когда переедешь на prstnk.ru — заменить домен в SEO

Сейчас canonical и og:url указывают на `https://bregeadmin.github.io/prstnk/` (текущий хостинг для теста). Когда переедешь на `prstnk.ru` — нужно заменить:

- В `index.html` и `lot.html` — все `https://bregeadmin.github.io/prstnk/` → `https://prstnk.ru/`
- В `sitemap.xml` — то же самое
- В `robots.txt` — Sitemap-строку

В большинстве редакторов это глобальная замена: Cmd+Shift+H в VS Code, или просто открыть файлы в TextEdit и Cmd+Opt+F.

## Деплой: студия в GitHub → витрина в России

Модель разделена на две части:

- **Студия** — GitHub-репозиторий. Здесь живут данные (`data/`), генератор (`build.py`) и админка. Сюда ты НЕ заходишь как посетитель — это «кухня».
- **Витрина** — российский хостинг под доменом `prstnk.ru`. Сюда заходят покупатели. Только готовые HTML + шрифты, ноль внешних зависимостей — в РФ открывается быстро, ничего не блокируется.

**Как это связано:** при любой правке (через CMS или JSON) GitHub Action собирает страницы и **сам заливает витрину на хостинг по FTP**. Тебе не надо ничего перезаливать вручную.

```
   правишь данные          GitHub Action               посетитель
  (CMS или JSON)   ──→   build.py + FTP-деплой   ──→   prstnk.ru (Beget)
     [GitHub]                 [GitHub]                  [Россия, быстро]
```

### Хостинг: рекомендую Beget

Почему Beget: простая панель, бесплатный SSL (Let's Encrypt), дёшево (~200₽/мес), российские дата-центры, поддерживает FTP-деплой из GitHub Action.
Альтернативы — Timeweb (так же хорош) или REG.RU (удобно, если домен уже там).

**Настройка (один раз):**

1. Регистрируешься на [beget.com](https://beget.com), берёшь тариф «Старт».
2. Панель → «Сайты» → создать сайт с доменом `prstnk.ru`. Удали дефолтный `index.html` Beget'а из `public_html/`.
3. Панель → «FTP» → создай FTP-аккаунт, **указав корневую папку = `public_html/` твоего домена**. Запиши: адрес сервера, логин, пароль.
4. В DNS домена пропиши NS-серверы Beget (инструкция в панели). Закажи бесплатный SSL.

### Подключить авто-деплой (один раз)

В GitHub-репозитории: **Settings → Secrets and variables → Actions → New repository secret**, добавь три секрета:

| Secret | Значение |
|---|---|
| `FTP_HOST` | адрес FTP-сервера (например `ftp.beget.com` или `prstnk.ru`) |
| `FTP_USERNAME` | FTP-логин из шага 3 |
| `FTP_PASSWORD` | FTP-пароль |

Также: **Settings → Actions → General → Workflow permissions → «Read and write»** (чтобы Action мог коммитить пересобранные страницы).

После этого любой коммит в `main` (хоть через CMS, хоть правка JSON) → Action соберёт и зальёт витрину на `prstnk.ru` автоматически. Пока секреты не заданы — Action просто собирает страницы, FTP-шаг пропускается (не падает).

> Если FTPS не подключается — в `.github/workflows/build.yml` поменяй `protocol: ftps` на `protocol: ftp`.

### Запасной вариант — ручная заливка

Если не хочешь возиться с Action: локально запусти `python3 build.py`, потом залей **всё, кроме** `data/`, `build.py`, `admin/`, `.github/`, `*.py`, `*.md` — в `public_html/` через файловый менеджер Beget. Витрина — это `*.html`, `prstnk.css`, `prstnk.js`, `fonts.css`, `fonts/`, `images/`, `favicon.svg`, `og-cover.svg`, `robots.txt`, `sitemap.xml`.

## Как открыть сайт локально (для проверки)

Просто двойным кликом по `index.html` — откроется в браузере. Все ссылки между страницами и шрифты работают без сервера, потому что используются относительные пути.

Можно также запустить локальный сервер для большего реализма (например, чтобы `robots.txt` отдавался по `/robots.txt`):

```bash
cd prstnk-site
python3 -m http.server 8080
# открыть http://localhost:8080
```

## Что устроено особенным образом

- **Все Telegram-кнопки генерятся в JS.** Ищи в HTML атрибут `data-tg-text="..."` — это текст, который появится в чате. JS соберёт ссылку формата `https://t.me/<TG_USER>?text=<encoded>`. Юзернейм меняется в одной строчке в `prstnk.js`.
- **Шрифты локальные.** Подключены через `fonts.css` из папки `fonts/`. Это сделано специально, потому что Google Fonts периодически замедляет Роскомнадзор — локальные шрифты загружаются с твоего же сервера и работают всегда.
- **Knак (фирменный знак PRSTNK)** меняет цвет центральной полосы при наведении на пункты навигации и на карточки работ — это реализовано в `prstnk.js`. Если будешь править — будь аккуратен, окна знака всегда контурные, никогда не закрашиваются.
- **Палитра, типографика и все размеры** хранятся как CSS-переменные в начале `prstnk.css` (раздел `:root`). Меняешь там — меняется по всему сайту.
- **Работы** сейчас нарисованы вручную в SVG прямо в HTML (стилизованные плейсхолдеры в духе риzo-печати). Когда появятся реальные сканы — каждую `<svg>...</svg>` заменишь на `<img src="works/название.jpg" alt="...">` и подправишь стили `.work__plate` если надо.
- **FAQ-блоки** реализованы через нативный `<details>/<summary>` — без JavaScript, работают даже если JS не загрузился.
- **Analytics-разметка.** На все кнопки, карточки и CTA поставлены атрибуты `data-analytics="..."` — единая схема именования. Когда подключишь Yandex.Metrica или GA — просто привяжешь обработчики к этим селекторам.
