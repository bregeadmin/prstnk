# ЁPRST — Материалы из блоков (track A)

**Дата:** 2026-05-30
**Проект:** prstnk.ru — магазин авторской печатной графики
**Статус:** на ревью

---

## Контекст

Журнал ЁPRST состоит из двух треков контента:

- **Track A — материалы** (этот спек): поток еженедельных материалов. Каждый материал самостоятелен, редактируется в админке, собирается из блоков. Это движок контента.
- **Track B — выпуск** (отдельно, позже): раз в месяц, штучная designed-вещь с обложкой и PDF. Верстается вручную (Artem + Claude). В этот спек **не входит**.

Track A проектируется первым, потому что выпуск собирается *из материалов* — без материалов выпуску нечего упаковывать. Заодно track A чинит сломанный рубрикатор на `/journal`.

Позиционирование прежнее: PRSTNK — **магазин**, который пилит контент про искусство, а не арт-институция.

---

## Терминология и разведение имён

⚠️ **Важно:** слово «материалы» в коде уже частично занято.

- Папка `data/materials/` сейчас хранит **ленту** (телеграм-посты: поля `tgText`, `url`, `tag: репортаж`). В `build.py` переменная `materials` питает секцию «Лента».
- Пользователь хочет называть **еженедельные статьи** «материалами».

**Решение (низкий риск, без churn на свежеотгруженном коде):**

| Сущность | Папка данных | Лейбл в админке |
|----------|--------------|-----------------|
| Еженедельные статьи (track A) | **`data/articles/`** (новая) | **«Материалы»** |
| Телеграм-лента (существует) | `data/materials/` (не трогаем) | **«Лента»** (переименовать лейбл) |

Папки данных пользователю не видны — он видит только лейблы в Sveltia. Поэтому внутреннее имя `data/articles/` при UI-лейбле «Материалы» допустимо. Внутренний рассинхрон (`data/materials` = лента) задокументирован здесь как осознанный компромисс, чтобы не переписывать отгруженный код журнала. Переименование `data/materials/ → data/lenta/` — за рамками (можно отдельной чисткой позже).

В тексте спека ниже **«материал» = статья из `data/articles/`**.

## Модель данных

### Материал — `data/articles/<slug>.json`

Одна статья = один JSON-файл (folder-коллекция Sveltia, как `artworks`/`artists`).

```json
{
  "slug": "kak-travyat-tsink",
  "rubric": "na-paltsakh",
  "title": "Как травят цинк: кислота и игла",
  "titleEm": "кислота и игла",
  "lead": "Офорт по шагам — от грунтовки доски до первого оттиска.",
  "cover": "/uploads/articles/tsink-hero.jpg",
  "author": "редакция ЁPRST",
  "date": "2026-05-14",
  "readMins": 7,
  "featured": false,
  "blocks": [
    { "type": "text", "text": "Офорт начинается не с рисунка, а с **подготовки доски**…" },
    { "type": "heading", "text": "Где кислота делает свою работу" },
    { "type": "photo", "image": "/uploads/articles/vanna.jpg", "caption": "Травильная ванна, 2024" },
    { "type": "split", "text": "Глубина борозды решает всё…", "image": "/uploads/articles/igla.jpg", "imageSide": "right" },
    { "type": "quote", "text": "Кислота не прощает спешки.", "cite": "Пётр Швецов" },
    { "type": "gallery", "images": ["/uploads/a1.jpg", "/uploads/a2.jpg", "/uploads/a3.jpg"] },
    { "type": "picks", "heading": "Офорты в продаже", "workIds": ["boloto-3", "sever-2", "tihiy-dvor"] },
    { "type": "embed", "provider": "youtube", "url": "https://youtu.be/…", "caption": "Печать на станке" }
  ]
}
```

**Поля материала:**

| Поле | Тип | Назначение |
|------|-----|------------|
| `slug` | строка | адрес `/journal/<slug>` |
| `rubric` | select | одна из 4 рубрик (slug) |
| `title` | строка | заголовок |
| `titleEm` | строка (опц.) | часть заголовка для курсивного алого выделения |
| `lead` | строка | лид-абзац (в превью и шапке) |
| `cover` | image (опц.) | обложка для карточек-превью и шапки материала |
| `author` | строка | по умолчанию «редакция ЁPRST», можно гостя |
| `date` | дата | дата публикации (сортировка ленты) |
| `readMins` | число | время чтения |
| `featured` | bool | поднимать ли в «герой» ленты/витрины |
| `blocks` | list (типизированный) | тело материала |

### Рубрики — фиксируются в коде (`build.py`), не в данных

4 рубрики, slug → лейбл → цвет:

| slug | лейбл | цвет | о чём |
|------|-------|------|-------|
| `razgovory` | Разговоры | алый `--c-alyi` | интервью с художниками |
| `na-paltsakh` | На пальцах | кобальт `--c-kobalt` | техники, как делается |
| `istoriya` | История | хвоя `--c-hvoya` | история графики, школ |
| `kak-smotret` | Как смотреть | фуксия `--c-fuxia` | как читать/понимать графику |

Словарь рубрик живёт в `build.py` как `RUBRICS = {...}` (slug → {label, color, color_var}). Используется для: select в админке, плашек-рубрик, рубрикатора, страниц рубрик.

## Блоки (8 типов)

Каждый блок — объект в массиве `blocks` с полем `type`. `build.py` содержит `render_block(block)` — диспетчер по типу. Утверждены в превью (`_blocks-preview.html`).

| type | поля | рендер |
|------|------|--------|
| `text` | `text` (markdown) | абзац(ы), Onest 19px/1.72 |
| `heading` | `text` | подзаголовок, Unbounded 28px |
| `photo` | `image`, `caption` (опц.) | фото на всю ширину колонки + подпись (моно) |
| `split` | `text`, `image`, `imageSide` (left/right) | 2 колонки: текст + фото рядом |
| `gallery` | `images[]` | сетка изображений (2–4 в ряд) |
| `quote` | `text`, `cite` (опц.) | крупный курсив Unbounded, алая черта слева |
| `picks` | `heading`, `workIds[]` | карточки работ из каталога (фото+цена+«в магазин») — связь с магазином |
| `embed` | `provider`, `url`, `caption` (опц.) | адаптивная встройка 16:9 (YouTube/Vimeo) |

**`picks`** берёт работы по id из существующего каталога (`data/artworks/`) — переиспользуем данные магазина, цену и ссылку на товар. Это стратегический блок: контент → продажа.

**Безопасность embed:** `provider` — select (`youtube`/`vimeo`), из `url` извлекается только id видео и подставляется в доверенный iframe-шаблон. Произвольный HTML не вставляется.

## Страницы

### 1. Страница материала — `/journal/<slug>.html`

Шапка (кикер-рубрика, H1 с `titleEm` курсивом, лид, мета: автор · время · дата) → тело из блоков → внизу «← Все материалы» и блок рубрики. Узкая колонка чтения (~680px), фото/галерея/picks могут быть шире. Шрифты — как на сайте. Эталон вёрстки — `_blocks-preview.html`.

### 2. «Все материалы» — ритм-лента

Длинная страница превью **из чередующихся модулей разной ширины и ориентации** (эталон — `_materials-feed-v3.html`). `build.py` проходит по материалам (сортировка по дате, новые сверху) и раскладывает их по циклу модулей:

**Цикл ритма (повторяется):**
1. `hero` — широкий: крупное фото слева + текст справа (1 материал)
2. `split` — пополам: 2 равные вертикальные карточки
3. `band` — широкая полоса: текст слева + панорамное фото справа (1 материал)
4. `split2` — пополам с вертикальным акцентом: высокая фича-колонка + стопка из 3
5. `textband` — без фото: плотный список коротких материалов

Логика раскладки: `build.py` берёт отсортированный список и «нарезает» его на модули по шаблону цикла (hero=1, split=2, band=1, split2=4, textband=3 → 11 материалов на цикл), затем повторяет. Если материалов меньше — недостающие модули цикла пропускаются (graceful degradation).

Низ страницы — кнопка «Загрузить ещё» (заглушка; бесконечный скролл — позже, отдельной задачей с JS).

URL: `/materials.html` (чистая ссылка `/materials`).

### 3. Страница рубрики — `/journal/<rubric>.html`

Те же ритм-модули, но только материалы одной рубрики. Заголовок страницы = лейбл рубрики в её цвете. Чинит рубрикатор: ссылки в верхней панели `/journal` ведут сюда. URL: `/journal/razgovory` и т.д.

⚠️ **Ограничение адресов:** материалы и рубрики делят неймспейс `/journal/<…>`. Slug материала **не должен** совпадать с одним из 4 slug'ов рубрик (`razgovory`, `na-paltsakh`, `istoriya`, `kak-smotret`). При реализации — проверка в `build.py`: если slug материала попал в зарезервированные, падать с понятной ошибкой при сборке.

### 4. `/journal` (витрина) — обновления

- **«Свежие материалы»** (редакционная сетка) теперь берёт данные из `data/articles/` (последние по дате), а не из статей выпуска.
- **Рубрикатор** оживает: ссылки ведут на `/journal/<rubric>`, счётчики `<sup>` = число материалов в рубрике.
- Секции «Лента» и «Выпуски» — без изменений (лента из `data/materials/`, выпуски из `data/issues/`).

## Интеграция build.py

Новые функции:
- `RUBRICS` — словарь рубрик (slug → label/color).
- `render_block(block)` — диспетчер: по `block["type"]` вызывает приватные рендереры (`_blk_text`, `_blk_photo`, …). Возвращает HTML.
- `render_article(article)` — шапка + `"".join(render_block(b) for b in blocks)` + футер. Пишет `/journal/<slug>.html`.
- `render_materials_index()` — ритм-лента всех материалов → `/materials.html`.
- `render_rubric_page(rubric)` — ритм-лента материалов рубрики → `/journal/<rubric>.html`.
- `_feed_rhythm(articles)` — общий хелпер: раскладывает список материалов по циклу модулей, возвращает HTML (используется и «Все материалы», и страницами рубрик).
- Обновить `render_journal_index()`: «Свежие материалы» и счётчики рубрикатора — из `articles`.

Загрузка: `articles = load_collection("articles")` (как `issues`/`materials`). Регистрация новых страниц в основном цикле сборки (рядом с генерацией остальных страниц). Чистые ссылки — через существующую `clean_links()`. Подключение CSS — `head(..., extra_css="journal.css")` (расширим существующий `journal.css`, новый файл не нужен).

CSS блоков и ритм-модулей добавляется в **существующий `journal.css`** (он уже подключается на журнальных страницах). Классы с понятными именами (`.bl-*` для блоков, `.feed-*` для модулей ленты).

## Перенос статей из выпуска 04

`issues/04.json` содержит статьи с полями `kicker/title/lead/body/pullquote/picks/bodyAfter/cta`. Переносим каждую в `data/articles/<slug>.json`:
- `kicker` → `rubric` (маппинг по смыслу: «Разговоры»→razgovory и т.д.)
- `title/lead/readMins` → одноимённые
- `body` (markdown) → разбить на блоки: основной текст → `text`/`heading`, `pullquote` → `quote`, `picks` → `picks`, `bodyAfter` → `text`, `cta` → пропустить/в текст
- проставить `date` (из периода выпуска), `author: "редакция ЁPRST"` или гость из `credits`

Это даёт «Все материалы» реальный контент на старте. Выпуск 04 как сущность остаётся в `data/issues/` (track B), но его статьи теперь полноценные материалы.

⚠️ Уточнить при реализации: оставить ли дублирование (статьи и в issue, и в articles) или issue начинает ссылаться на slug'и материалов. Для track A достаточно перенести — связь issue↔материалы решается в track B.
